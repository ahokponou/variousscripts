# 📜 CA Helper Scripts – Quick Reference

A tiny, self‑contained suite of Bash scripts that lets you:

1. Create your own private Certificate Authority (CA)
2. Install the root certificate on a Linux system
3. Generate a CSR (with SANs) for any service
4. Sign that CSR with your CA

Everything lives in a single directory and uses only ` openssl ` and standard Linux utilities.

## 📂 Repository layout

```txt
|-- generate-ca.sh      # Build a new CA (key + self‑signed root cert)
|-- install-ca.sh       # Copy the root cert into the system trust store (linux only)
|-- request-cert.sh     # Create a private key + CSR (incl. SANs)
|-- sign-cert.sh        # Sign the CSR with the CA, producing a leaf cert
```

Each script creates a predictable folder structure under the directory where you run them:

```txt
<CA_NAME>/
|
|-- private/            # Private keys (CA key, leaf keys)
|-- csr/                # Certificate Signing Requests
|-- certs/              # Issued certificates (root + leaf)
|-- cnf/                # Temporary OpenSSL config files (used for SANs)
```

## 🛠 Prerequisites

| Requirement | Why it matters |
| --- | --- |
| Linux/macOS (any POSIX‑compatible shell) | Scripts rely on Bash built‑ins (read, set -e, etc.). |
| OpenSSL (≥ 1.1.1) | Core crypto operations – key generation, CSR creation, signing. |
| sudo (for ` install-ca.sh `) | Needed to write to the system CA directory (` /usr/local/share/ca-certificates `). |
| Optional: ` update-ca-certificates ` (Debian/Ubuntu) or equivalent (` update-ca-trust ` on RHEL/Fedora) | Makes the OS trust certificates signed by your CA. |

> Tip: On macOS you can still run the scripts, but you’ll need to manually add the root cert to the Keychain (security add-trusted-cert ...).

## 🚀 Quick start – step‑by‑step

Below is the typical workflow for a single CA named ` myCA ` and a leaf certificate for a service called ` myservice `.

> **All commands assume you are in the directory that contains the four scripts.**

### 1. Create the CA

```shell
./generate-ca.sh
```

Prompt: "Enter Certificate Authority (CA) name:" → type ` myCA `

Result:

```txt
myCA/
|-- private/myroot.key          # Encrypted RSA private key (DES‑3)
|-- certs/myroot.pem            # Self‑signed root certificate (valid 10 y)
```

### 2. Install the root cert (Linux only)

```shell
./install-ca.sh myCA
```

What happens:

- Copies ` myCA/certs/myroot.pem ` -> ` /usr/local/share/ca-certificates/myCA.crt `
- Runs ` sudo update-ca-certificates ` so the OS trusts anything signed by this CA.

> If you’re on a non‑Debian distro, replace the copy target with the appropriate location (` /etc/pki/ca-trust/source/anchors/ `) and run ` sudo update-ca-trus `.

### 3. Generate a CSR for your service

```shell
./request-cert.sh myservice myCA
```

Prompts:
- "Enter comma‑separated SANs ..." -> e.g.

` DNS:myservice.local,DNS:example.com,IP:192.168.1.10 `

Result:

```txt
myCA/
|-- private/myservice.key       # Unencrypted leaf private key
|-- csr/myservice.csr           # CSR containing the CN and SANs
|-- cnf/myservice.cnf           # Temp OpenSSL config used for the CSR
```

### 4. Sign the CSR with your CA

```shell
./sign-cert.sh myservice myCA
```

Result:

```txt
myCA/certs/myservice.crt      # Leaf certificate (valid 825 days)
```

You can now configure your server/software to use ` myservice.key ` and ` myservice.key `. Because the root CA is trusted (step 2), clients will accept the connection without warnings.

## 🔧 Customisation tips

| Area | What you might change | How |
| --- | --- | --- |
| Key size / algorithm | Use RSA 4096 or ECC (e.g., ` prime256v1 `). | Edit the ` openssl genrsa ` line in ` generate-ca.sh ` or replace it with ` openssl ecparam -genkey -name prime256v1 `. |
| Passphrase handling | Remove the ` -des3 ` flag for an unencrypted CA key (use with caution). | Delete ` -des3 ` from the ` openssl genrsa ` command. |
| Validity periods | Shorter/longer lifetimes for root or leaf certs. | Change ` -days 3650 ` (CA) or ` -days 825 ` (leaf) in the respective scripts. |
| Additional extensions | 	Add ` keyUsage `, ` extendedKeyUsage `, etc. | Extend the ` [req_ext] ` block in the temporary ` .cnf ` files (generated by ` request-cert.sh `).|
| Cross‑platform install | 	macOS, Windows, or containerised environments. | Replace the copy‑and‑update logic in ` install-ca.sh ` with platform‑specific commands (e.g., security ` add-trusted-cert ` on macOS). |



## ❗️ Safety & Best Practices

1. **Protect the CA private key** – It signs every leaf certificate. Keep it offline or encrypted if possible.
2. **Limit distribution** – Only install the root cert on machines that truly need to trust the issued leaf certs.
3. **Rotate regularly** – When the CA key expires or is compromised, generate a fresh CA and re‑issue leaf certs.
4. **Audit SANs** – Ensure the SAN list you provide matches the hostnames/IPs your services actually use.
